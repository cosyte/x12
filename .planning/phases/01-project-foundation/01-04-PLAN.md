    ```yaml
    name: CI

    on:
      push:
        branches: [main]
      pull_request:
        branches: [main]

    permissions:
      contents: read

    concurrency:
      group: ci-${{ github.workflow }}-${{ github.ref }}
      cancel-in-progress: true

    jobs:
      verify:
        name: verify (node ${{ matrix.node }})
        runs-on: ubuntu-latest
        strategy:
          fail-fast: false
          matrix:
            node: ["18", "20", "22"]
        steps:
          - uses: actions/checkout@v4

          - name: Install pnpm
            uses: pnpm/action-setup@v4
            with:
              # Mirror the `packageManager` pin from package.json; do NOT specify
              # `version:` here if package.json `packageManager` is set, as
              # pnpm/action-setup@v4 reads it automatically. Specify explicitly
              # only if sibling project does.
              run_install: false

          - name: Setup Node ${{ matrix.node }}
            uses: actions/setup-node@v4
            with:
              node-version: ${{ matrix.node }}
              cache: pnpm

          - name: Install dependencies
            run: pnpm install --frozen-lockfile

          - name: Typecheck
            run: pnpm typecheck

          - name: Lint
            run: pnpm lint

          - name: Format check
            run: pnpm format:check

          - name: Test
            run: pnpm test

          - name: Build
            run: pnpm build

          - name: Verify dual ESM + CJS exports resolution
            run: pnpm run verify:exports
    ```
    Notes:

    - `concurrency` cancels superseded runs on the same branch to save Actions minutes.
    - The order â€” typecheck â†’ lint â†’ format:check â†’ test â†’ build â†’ verify:exports â€” matches the inner-loop chain a developer runs locally (early gates fail fast).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner â†’ repo contents | Workflow checks out code and runs npm/pnpm scripts. Has read-only `contents` permission per workflow config. |
| GitHub Actions runner â†’ npm registry | `pnpm install --frozen-lockfile` resolves devDeps against the lockfile (no network drift). |
| Workflow definition file â†’ external actions | `actions/checkout`, `actions/setup-node`, `pnpm/action-setup` are third-party JavaScript actions executed inside the runner. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-10 | Tampering | GitHub Actions third-party action supply chain | mitigate | Pin to major-version tags (`@v4`) from well-known publishers (actions/*, pnpm/*). Pinning to SHA is a stronger mitigation but trades agility; the sibling project's choice governs (mirror exactly). Document the policy in SUMMARY. |
| T-01-11 | Elevation of Privilege | Workflow permissions overscope | mitigate | Top-level `permissions: contents: read` enforces least privilege. No `write`, `packages`, `id-token`, or `pull-requests` scopes granted in this phase (release tooling lands in Phase 8 and may need additional scopes â€” that is Phase 8's threat surface, not Phase 1's). |
| T-01-12 | Tampering | `pnpm install` drift between developer and CI | mitigate | `--frozen-lockfile` flag fails CI if `pnpm-lock.yaml` is out of sync; committing the lockfile locks the dep tree. |
| T-01-13 | Information Disclosure | Workflow logs leak secrets | accept | No secrets are referenced in this workflow. If Phase 8 adds publish workflow with `NPM_TOKEN`, that workflow gets its own threat model. |
| T-01-14 | Spoofing | Untrusted PR running with elevated privileges | mitigate | `pull_request` events from forks run with read-only token by default; no `pull_request_target` event used; no secrets exposed; CI is build-only. |

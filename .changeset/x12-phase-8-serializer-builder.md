Phase 8 — spec-clean serializer + general interchange builder (the emit half of the parser lands). Two new public surfaces close the read↔write loop:

**`serializeX12(interchange, opts?)`** turns any parsed `X12Interchange` back into an X12 byte stream. Default mode is byte-faithful — the interchange is reconstructed purely from the verbatim `.raw` strings the parser preserved (ISA + terminator, then each TA1 / GS / transaction segment / GE / IEA terminator-joined, then any `trailingBytes`), so for a Tier-1 (spec-clean, no-CRLF) input it reproduces the source bytes exactly: the idempotency fixed point `serialize(parse(s)) === s`. With `{ specClean: true }` the serializer ALSO reconciles the envelope — SE-01 (segment count), GE-01 (transaction count), IEA-01 (group count), and the ISA-13↔IEA-02 / GS-06↔GE-02 / ST-02↔SE-02 control-number pairs — surfacing every mismatch via `opts.onWarning` and NEVER silently correcting it. Corrected counts are emitted only when the caller ALSO opts in with `{ recomputeCounts: true }`; control NUMBERS are identity, not derived, so they are NEVER rewritten, only flagged. The new `X12_SEGMENT_COUNT_MISMATCH` warning code is a serializer-only diagnostic (the parser never validated SE-01) — the registry expands 21 → 22 (additions-only), and its factory carries bounded positional + numeric metadata only, never an element value (H-PHI invariant).

**`buildInterchange(spec)`** is the general-purpose, segment-level builder: given an `InterchangeSpec` (ISA identity + groups + per-transaction body segments) it owns every envelope mechanic a caller should never hand-roll — the 106-byte fixed-width ISA layout, the GS/GE/SE/IEA control segments, and the SE-01 / GE-01 / IEA-01 counts — then round-trips the result back through `parseX12` so the returned interchange is bit-identical to the parsed form every other helper consumes (and any internal builder bug surfaces as a warning, not silent corruption). Active delimiters inside body element values are escaped via the `?` release character. Structurally impossible specs are REFUSED with a typed `X12BuildError` (`X12_BUILD_INVALID_SPEC`): an over-long ISA-13 control number, a body segment with no segment id. New exports: `serializeX12`, `SerializeOptions`, `buildInterchange`, `InterchangeSpec`, `FunctionalGroupSpec`, `TransactionSetSpec`, `SegmentSpec`, `X12BuildError`, `X12_BUILD_ERROR_CODES`, `X12BuildErrorCode`, and the `segmentCountMismatch` warning factory.

Round-trip goldens lock the emit surface across all v1 transactions: 13 committed `test/fixtures/golden/<name>.edi` files (835, 837P/I/D, 271, 277, 277CA, 278 request + response, 820, 834, 999, TA1) regenerated by `test/scripts/gen-serialize-goldens.ts`, asserting `serializeX12(parseX12(fixture))` reproduces the golden byte-for-byte and that re-serializing a golden is a no-op. A `roundTripProperty` run (300 cases) over spec-clean generated interchanges plus a builder property (200 cases) assert serialize idempotency and that the builder never emits a self-inconsistent envelope (zero warnings on re-parse, serialize fixed point).

This phase also surfaced — and fixed — four latent fixture defects the lenient parser never caught (it validates GE-01 / IEA-01 / control pairs but not SE-01): hand-miscounted SE-01 in `837i-canonical` (30→33), `837d-canonical` (25→26), and `999-accept` (5→6), plus a GS-06/GE-02 control mismatch in `278-response` (GS-06 2→1). The new serializer reconciliation caught all four — a real accuracy-gate win.

Known limitations (deferred to a follow-up): domain per-transaction builders (`build835` / `build837P/I/D` / `build271` / etc. — the safety-critical emit code that enforces per-TR3 balance and certification invariants) are NOT in this phase; `serializeX12` + `buildInterchange` are the general envelope-level surface they will layer on top of.

Phase 7 — 278 health-care services review + 834 benefit enrollment + 820 premium payment. Four new public read-side helpers extract typed models from a parsed transaction set, rounding out the v1 HIPAA transaction scope: `get278Request(delimiters, tx)` (TR3 `005010X217`) and `get278Response(delimiters, tx)` (TR3 `005010X216`); `get820Payments(delimiters, tx)` (TR3 `005010X218`); and the streaming pair `get834Header(delimiters, tx)` + `get834Enrollments(delimiters, tx)` (TR3 `005010X220A1`). Each entry point returns `undefined` (or, for the stream, yields nothing) only on a mis-routed call (wrong `ST-01`); every recoverable deviation surfaces as a warning, never a throw.

**The certification decision and maintenance type are the safety-critical surfaces.** In a 278 response the `HCR-01` action code (certified / not-certified / pended / modified) is captured verbatim onto each event / service review — the walker NEVER infers a certification outcome. In an 834 the `INS-03` / `HD-01` maintenance type (X12 0875) is always preserved verbatim and never mapped to an inferred action; an unrecognized code raises `X12_834_UNKNOWN_MAINTENANCE_TYPE` on the affected member only. The 278 HL spine `20 → 21 → 22 → 23` is validated for parent-pointer integrity via the shared `validateHl`; the `EV` (patient-event) and `SS` (service) levels are intentionally tolerant (omitted from the expected-parent map, since clearinghouses vary on where they attach).

**834 streaming.** `get834Enrollments` is an `AsyncIterable<X12Enrollment>` yielding one decoded member per `INS` loop, so a consumer processing a hundreds-of-MB roster holds one member at a time rather than the whole decoded result set — verified by a streaming property test over a programmatically generated 10MB+ synthetic 834 with early-break. (Honest limitation: the v1 source still parses the file into `tx.segments` up front; a true file→iterator parser is a v2 item.)

All monetary fields decode as `X12Decimal` (BigInt-exact, never `parseFloat`): the 820 BPR premium total, RMR amounts-paid / amounts-due and ADX adjustments; the 834 HD-loop AMT premiums. The 820 surfaces the BPR payment header, TRN reassociation traces, premium-receiver (`N1*PE`) + remitter (`N1*PR` / `N1*RM`) parties with addresses, and both organization-summary (`ENT`) and bare-`NM1` individual remittance loops with their RMR open items, DTM dates, and ADX adjustments. Twelve dogfooded `LoopSpec` artifacts ship through the public `defineLoopSpec()` (6 × 278 HL hierarchy, 3 × 820, 3 × 834). The warning registry gains `X12_834_UNKNOWN_MAINTENANCE_TYPE` (additions-only); its factory shape-validates the echoed code (H-PHI invariant — the message never carries member PHI). The HI diagnosis qualifier → code-system provenance registry and `X12Hl` HL primitive are reused from Phases 5–6. Synthetic fixtures across all three surfaces (278 request / response / comprehensive / edge; 820 canonical / edge / loop; 834 canonical / edge), unit tests, and the 834 streaming property.

Known limitations (deferred): 278 detailed 2010E provider sub-loops (rendering / attending / operating), SV1/SV2/SV3 service-line detail, PWK attachments, HSD service-delivery, and CRC condition codes are preserved on `tx.segments` but not destructured; 820 RMR/REF inside an open remittance loop and non-enumerated optional segments (NTE / CUR / PER) are likewise preserved verbatim; 834 non-`IL` member roles (custodial parent, responsible person) and Loop 2750 reporting categories stay on `tx.segments`. Serialization (build-side) for all three is Phase 8.

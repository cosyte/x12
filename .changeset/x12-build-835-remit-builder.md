Domain builder — `build835` (005010X221A1 Health Care Claim Payment/Advice). The first per-transaction emit helper layers the safety-critical TR3 invariants on top of the Phase 8 general builder, mirroring the pure-function `build999` / `buildTA1` pattern: it NEVER auto-sends, NEVER opens a socket, NEVER touches the filesystem.

**`build835(spec)`** assembles a complete `X12Interchange` — one GS..GE functional group (GS-01 `HP`) wrapping one ST..SE 835 transaction set (ST-03 `005010X221A1`) — from a typed `Build835Spec` whose monetary fields are `X12Decimal` throughout (BigInt-exact, never `parseFloat`). Segments emit in TR3 loop order (BPR → TRN* → Loop 1000A/1000B parties → LX → Loop 2100 claims with CLP / CAS / NM1 / DTM / REF / AMT / LQ → Loop 2110 service lines → PLB), and the result round-trips through `parseX12` so a balanced spec is reproduced by `get835` field-for-field. Composite elements (CLP-08, SVC-01, SVC-06, PLB) escape each component then join with the raw component separator — the envelope is emitted inline (not delegated to `buildInterchange`) precisely to avoid double-escaping a pre-composed element. Consecutive same-group CAS adjustments and same-provider/period PLB adjustments pack into one segment (≤ 6 triples / pairs each), mirroring the read side's flatten; PLB amounts carry the raw EDI sign (`BPR-02 == Σ(CLP-04) − Σ(PLB)`).

**Refusal, not silent corruption.** Where the lenient read side (`get835`) only WARNS on an out-of-balance payer artifact, the builder REFUSES via a typed `Remit835BuildError`. It reuses the authoritative read-side validators (`checkServiceLineBalance` / `checkClaimBalance` / `checkRemitTotalBalance`) against a materialized read model so the emit guard and the parse warning share one source of truth — emitting an out-of-balance 835 would tell a downstream cash-poster that money math holds when it does not. Error codes: `X12_835_BUILD_BALANCE_MISMATCH` (any §1.10.2 invariant violated — service line, claim, or top-of-remit) and `X12_835_BUILD_INVALID_SPEC` (no TRN trace, a claim with an empty CLP-01 patient-control number, an over-long ISA-13 control number). The thrown message carries numeric totals only — never a patient-control number or member id (PHI discipline).

New public exports: `build835`, `Remit835BuildError`, `REMIT_835_BUILD_ERROR_CODES`, `Remit835BuildErrorCode`, and the `Build835Spec` type tree (`Build835EnvelopeSpec`, `Build835PaymentSpec`, `Build835TraceSpec`, `Build835PartySpec`, `Build835AddressSpec`, `Build835ReferenceSpec`, `Build835ContactSpec`, `Build835PersonSpec`, `Build835ProviderSpec`, `Build835AdjustmentSpec`, `Build835RemarkSpec`, `Build835AmountSpec`, `Build835ServiceLineSpec`, `Build835ClaimSpec`, `Build835ProviderAdjustmentSpec`). Verify gate green: typecheck, lint, format, phi-scan, coverage (per-dir ≥ 90, build-835.ts at 94.6% branches), build, attw, verify:exports.

**Parser fix surfaced by the round-trip review.** Implementing `build835` exposed a latent bug in the envelope tokenizer: `splitSegments` used a naive `indexOf` for the segment terminator and so split mid-value on a `?`-release-escaped terminator (`?~`, as emitted by `escapeRelease`) — truncating the segment, injecting a phantom empty segment, and silently corrupting the value, even though the element splitter (`splitWithRelease`) had always been release-aware. `splitSegments` now mirrors the element-splitter scan so an escaped terminator stays inside its value; a degenerate delimiter set where the terminator IS the release character falls back to the literal scan, preserving prior behaviour. This underpins both the new `build835` round-trip and the Phase 8 `serialize(parse(s)) === s` fixed point. Regression tests added at both the parser level (`parser-envelope`) and the builder level (`build835` round-trips a value carrying `~` / `*` / `:`).

Known limitation (deferred to chained follow-ups): the remaining domain builders (`build837P/I/D`, `build271`, `build277`, `build278`, `build820`, `build834`) layer on this same general surface and are NOT in this change.

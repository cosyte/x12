Domain builders — `build837P` / `build837I` / `build837D` (005010 837 Health Care Claim: Professional `X222A2`, Institutional `X223A3`, Dental `X224A2`). The claim-submission emit counterpart to `get837Claims`, layered on the Phase 8 general builder and mirroring the pure-function `build835` pattern: they NEVER auto-send, NEVER open a socket, NEVER touch the filesystem.

**`build837P/I/D(spec)`** each assemble a complete `X12Interchange` — one GS..GE functional group (GS-01 `HC`) wrapping one ST..SE 837 transaction set (ST-03 per variant) — from a typed `Build837Spec` whose monetary fields are `X12Decimal` throughout (BigInt-exact, never `parseFloat`). Segments emit in TR3 loop order (BHT → Loop 1000A/1000B parties → Loop 2000A/B/C HL spine → Loop 2300 claim: CLM / DTP / HI / NTE / AMT / REF / 2310 providers / 2320 other subscribers → Loop 2400 service lines: LX / SVx / DTP / LIN+CTP / TOO / NTE / AMT / REF / 2420 providers / 2430 SVD+CAS+DTP), and the result round-trips through `parseX12` so a well-formed spec is reproduced by `get837Claims` field-for-field. The envelope emits inline (not delegated to `buildInterchange`) so a pre-composed composite element is never double-escaped; one HI composite emits per HI segment so the read side's per-bucket diagnosis/procedure order is preserved; consecutive same-group line-adjudication CAS triples pack into one CAS segment (≤ 6 each).

**The HL spine is computed, never caller-supplied.** The builder OWNS the 837's safety primitive: it computes every HL-01 id (sequential within the transaction), HL-02 parent pointer (20 → 22 → 23), and HL-04 has-child flag from the nested billing-provider → subscriber → (claims | patient) tree. A structurally inconsistent hierarchy is therefore *unrepresentable*, and the SE-01 segment count is correct by construction.

**Refusal, not silent corruption.** Where the lenient read side (`get837Claims`) only WARNS on a broken HL parent pointer, the builder REFUSES a structurally impossible spec via a typed `Claim837BuildError` rather than emit a hierarchy a downstream payer would have to repair (or that would mis-route a claim to the wrong subscriber). Error codes: `X12_837_BUILD_INVALID_HIERARCHY` (no billing providers, a billing provider with no subscriber, a subscriber with neither a direct claim nor a dependent patient, a dependent patient with no claim) and `X12_837_BUILD_INVALID_SPEC` (empty `claimId`, a claim with no service line, a service line whose `variant` does not match the builder, an empty procedure / revenue code, an over-long control number). The thrown message carries structural locators only (`billing[0].subscriber[0].claim[0]`, level codes, counts) — never the `claimId` (patient-account number) or a member id (PHI discipline).

New public exports: `build837P`, `build837I`, `build837D`, `Claim837BuildError`, `CLAIM_837_BUILD_ERROR_CODES`, `Claim837BuildErrorCode`, and the `Build837Spec` type tree (`Build837EnvelopeSpec`, `Build837EntitySpec`, `Build837AddressSpec`, `Build837ContactSpec`, `Build837ReferenceSpec`, `Build837DateSpec`, `Build837HiCodeSpec`, `Build837NoteSpec`, `Build837AmountSpec`, `Build837AdjustmentSpec`, `Build837AdjudicationSpec`, `Build837DrugSpec`, `Build837ToothSpec`, `Build837ServiceLineBaseSpec`, `Build837ServiceLineProfessionalSpec`, `Build837ServiceLineInstitutionalSpec`, `Build837ServiceLineDentalSpec`, `Build837ServiceLineSpec`, `Build837OtherSubscriberSpec`, `Build837ClaimSpec`, `Build837SubscriberInfoSpec`, `Build837PatientSpec`, `Build837SubscriberSpec`, `Build837BillingProviderSpec`, `Build837Spec`). Verify gate green: typecheck, lint, format, phi-scan, coverage (per-dir ≥ 90), build, attw, verify:exports.

Known limitation (deferred to chained follow-ups): claim-/line-level provider addresses (Loop 2310/2420 N3/N4) are a documented read-side limitation — the NM1 fields round-trip, the address does not. The remaining domain builders (`build271`, `build277`, `build278`, `build820`, `build834`) layer on this same general surface and are NOT in this change.
